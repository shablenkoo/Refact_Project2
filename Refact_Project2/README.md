

## 📖 Огляд проєкту
Цей проєкт є реалізацією мікросервісу для керування QR-кодами (генерація, сканування, історія). Побудований з акцентом на **Чисту архітектуру**, **SOLID** та **автоматизоване тестування**.


---

## 🛠 Етапи реалізації

### Етап 1. Вимоги та UML-моделювання
- **Проблема:** Деталізація логіки нарахування штрафів (блокування QR при порушенні лімітів) та пріоритетність черги на резервування.
- **Сценарії:** Видача (сканування читачем), повернення (сканування бібліотекарем), блокування.
- **UML:** Діаграми прецедентів, класів та предметної області знаходяться в `/docs/diagrams/`.

### Етап 2. Архітектура та SOLID
- **Clean Architecture:** Код розділений на шари: `Models`, `Services`, `In-Memory Storage`.
- **GoF Patterns:**
  - **Strategy:** Для зміни формату генерації (PNG, Metadata) без зміни сервісу.
  - **Observer:** Для автоматичного логування історії та аналітики.
- **SOLID:** Кожен клас має одну відповідальність, залежності інвертовані через інтерфейси.

### Етап 3. ШІ-Архітектура (AI-Driven)
- Налаштовані глобальні правила `.cursorrules` для автономних агентів.
- Додано контекстні файли в `.cursor/rules/`: `architecture.md` та `testing_strategy.md`.

### Етап 4. Бізнес-логіка та Тестування
- **Тести:** Реалізовано **210+ тестів** (Unit та Integration).
- **Покриття:** > 70% згідно зі звітами JaCoCo/Coverage.py.
- **Edge Cases:** Обробка порожніх даних, неіснуючих ID та лімітів доступу.

### Етап 5. CI/CD та DevOps
- **Пайплайн:** GitHub Actions виконує Build -> Test -> Coverage -> SonarCloud.
- **Артефакти:** HTML-звіти про покриття генеруються після кожного коміту та доступні для завантаження в Actions.
- **Quality Gate:** Заборона злиття PR при покритті < 70%.
